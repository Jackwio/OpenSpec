# CLI 存檔指令說明

## 目的
archive 指令將已完成的變更從活動變更目錄移至具有基於日期的命名的存檔資料夾，遵循 OpenSpec 約定。

## 命令語法
```bash
openspec archive [change-name] [--yes|-y]
```

選項：
- `--yes`, `-y`：跳過確認提示（用於自動化）
## 要求
### 要求：更改選擇

該命令應支援互動式和直接更改選擇方法。

#### 場景：互動選擇

- **何時** 未提供更改名稱
- **然後** 顯示可用變更的互動式清單（不包括存檔/）
- **並且**允許使用者選擇一個

#### 場景：直接選擇

- **何時** 提供更改名稱
- **然後** 直接使用該更改
- **並且**驗證它存在

### 要求：任務完成檢查

該命令應在歸檔之前驗證任務完成狀態，以防止過早歸檔。

#### 場景：發現未完成的任務

- **何時** 發現不完整的任務（標記為 `- [ ]`)
- **然後** 向使用者顯示所有未完成的任務
- **並且**提示確認繼續
- **並且**為了安全起見預設為“否”

#### 場景：所有任務完成

- **何時** 所有任務均已完成或不存在任何tasks.md
- **然後** 繼續存檔而不提示

### 需求：存檔過程

歸檔操作應遵循結構化流程以安全地將變更移至歸檔。

#### 場景：執行歸檔

- **何時** 歸檔更改
- **然後** 執行以下步驟：
  1. 如果 archive/ 目錄不存在，則建立它
  2. 產生目標名稱為 `YYYY-MM-DD-[change-name]` 使用目前日期
  3. 檢查目標目錄是否已存在
  4. 根據變更的未來狀態規格更新主要規格（請參閱下面的規格更新流程）
  5. 將整個變更目錄移至存檔位置

#### 場景：存檔已存在

- **何時** 目標存檔已存在
- **然後** 失敗並顯示錯誤訊息
- **且**不覆蓋現有存檔

#### 場景：成功歸檔

- **何時** 移動成功
- **然後** 顯示成功訊息以及存檔名稱和更新規格列表

### 要求：規格更新流程

在將變更移至存檔之前，該命令應將增量變更套用至主要規格以反映部署的實際情況。

#### 場景：應用增量更改

- **何時** 使用基於增量的規範歸檔變更
- **THEN** parse and apply delta changes as defined in openspec-conventions
- **並且**在應用之前驗證所有操作

#### 場景：驗證增量更改

- **何時** 處理增量更改
- **然後** 依照 openspec-conventions 中指定的方式執行驗證
- **並且** 如果驗證失敗，則顯示特定錯誤併中止

#### 場景：衝突偵測

- **何時** 應用增量會建立重複的需求標頭
- **然後** 中止並顯示衝突的錯誤訊息
- **並**建議手動解決

### 要求：確認行為

規範更新確認應在應用變更之前提供清晰的可見性。

#### 場景：顯示確認

- **何時** 提示確認
- **然後** 顯示清晰的摘要，其中顯示：
  - 將建立哪些規範（新功能）
  - 哪些規格將會更新（現有功能）
  - 每個規範的來源路徑
- **和** 將確認提示格式化為：
  ```
  The following specs will be updated:
  
  NEW specs to be created:
    - cli-archive (from changes/add-archive-command/specs/cli-archive/spec.md)
  
  EXISTING specs to be updated:
    - cli-init (from changes/update-init-command/specs/cli-init/spec.md)
  
  Update 2 specs and archive 'add-archive-command'? [y/N]:
  ```
#### 場景：處理確認回應

- **何時** 等待使用者確認
- **那麼** 為了安全起見預設為「否」（需要明確的「y」或「是」）
- **並且**跳過確認 `--yes` 或者 `-y` 提供了標誌

#### 場景：用戶拒絕確認

- **何時** 用戶拒絕確認
- **然後** 中止整個歸檔操作
- **並且**顯示訊息：“存檔已取消。未進行任何更改。”
- **並且** 以非零狀態代碼退出

### 要求：錯誤條件

該命令應優雅地處理各種錯誤情況。

#### 場景：處理錯誤

- **何時**發生錯誤
- **THEN** handle the following conditions:
  - 缺少 openspec/changes/ 目錄
  - 未找到更改
  - 存檔目標已存在
  - 檔案系統權限問題

### 要求：跳過規格選項

歸檔命令應支援 `--skip-specs` 跳過所有規範更新操作並直接進行歸檔的標誌。

#### 場景：使用標誌跳過規範更新

- **何時**執行 `openspec archive <change> --skip-specs`
- **然後** 跳過規範發現和更新確認
- **並且**直接繼續將變更移至存檔
- **並** 顯示一則訊息，指示已跳過規格

### 要求：非阻塞確認

當使用者拒絕規範更新時，歸檔操作應繼續進行，而不是取消整個操作。

#### 場景：使用者拒絕規格更新確認

- **何時** 使用者拒絕規格更新確認
- **然後** 跳過規格更新
- **並且** 繼續歸檔操作
- **且** 顯示成功訊息，表示規格未更新

### 要求：顯示輸出

該命令應提供有關增量操作的明確回饋。

#### 場景：顯示增量應用程式

- **何時**套用增量更改
- **然後** 顯示每個規格：
  - 增加的要求數量
  - 修改的要求數量
  - 刪除的要求數量
  - 重命名的需求數量
- **AND** 使用 openspec-conventions 中定義的標準輸出符號 (+ ~ - →)：
  ```
  Applying changes to specs/user-auth/spec.md:
    + 2 added
    ~ 3 modified
    - 1 removed
    → 1 renamed
  ```

### 需求：存檔驗證

歸檔命令應在應用變更之前驗證變更以確保資料完整性。

#### 場景：預先歸檔驗證

- **何時**執行 `openspec archive change-name`
- **然後** 先驗證變更結構
- **並且**僅在驗證通過時繼續
- **並且** 如果失敗則顯示驗證錯誤

#### 場景：強制存檔而不驗證

- **何時**執行 `openspec archive change-name --no-validate`
- **然後** 跳過驗證（不安全模式）
- **並且** 顯示有關跳過驗證的警告

## 為什麼要做出這些決定

**互動式選擇**：減少打字並幫助使用者檢視可用的更改
**任務檢查**：防止意外歸檔不完整的工作
**日期前綴**：保持時間順序並防止命名衝突
**不覆蓋**：保留歷史檔案並防止資料遺失
**存檔前的規格更新**：主目錄中的規格代表當前的現實；當部署和存檔變更時，其未來狀態規範將成為新的現實，並且必須取代主要規範
**規範更新確認**：提供將要更改的內容的可見性，防止意外覆蓋，並確保用戶在修改規範之前瞭解影響
**--自動化的 yes 標誌**：允許 CI/CD 管道在沒有互動式提示的情況下存檔，同時預設保持手動使用的安全性
