# opsx-onboard-skill 規範

## 目的
定義 `/opsx:onboard` 用於指導使用者在其真實程式碼庫上完成端到端 OpenSpec 工作流程的行為。

## 要求
### 需求：OPSX 船上技能

系統應提供 `/opsx:onboard` 透過旁白和真實的程式碼庫工作引導使用者完成第一個完整的 OpenSpec 工作流程週期的技能。

#### 場景：技能調用

- **何時** 用戶調用 `/opsx:onboard`
- **那麼** 代理程式檢查 OpenSpec 是否已初始化
- **AND** 如果未初始化，則提示使用者執行 `openspec init` 第一的
- **並且** 如果已初始化，則繼續進行入職流程

#### 場景：歡迎和期望

- **何時** 入職開始
- **然後** 代理顯示歡迎訊息，解釋將會發生什麼
- **AND** 設定約 15 分鐘持續時間的預期
- **AND** 解釋了工作流程階段：探索 → 新建 → 工件 → 應用 → 存檔

### 要求：任務建議的程式碼庫分析

此技能應分析使用者的程式碼庫以建議適當範圍的啟動任務。

#### 場景：程式碼庫掃描

- **何時** 入職進入任務選擇階段
- **然後** 代理掃描程式碼庫以尋找小的改進機會
- **AND** 尋找：TODO/FIXME 註解、缺少錯誤處理、未經測試的函數、過時的依賴項、類型：TypeScript 中的任何內容、生產程式碼中的 console.log、缺少輸入驗證
- **並且** 檢查最近的 git 提交以獲取當前工作的上下文

#### 場景：任務建議演示

- **何時** 代理程式分析了程式碼庫
- **然後** 代理提出 3-4 個具體任務建議以及範圍估計
- **和** 每個建議包括：任務描述、估計範圍（文件/行）、為什麼它是一個好的入門者
- **AND** 提供使用者指定自己任務的選項

#### 場景：範圍護欄

- **何時** 使用者選擇或描述太大的任務
- **然後** 代理程式輕輕地重定向到較小的範圍
- **AND** 建議分解或延遲大型任務
- **並且**提供適當大小的替代方案

### 要求：探索階段演示

該技能應在建立更改之前簡要演示探索模式。

#### 場景：簡要探索演示

- **何時** 選擇任務
- **那麼** 代理簡要演示 `/opsx:explore` 透過調查相關代碼
- **AND** 解釋探索模式是為了先思考再做
- **並且** 保持此階段較短（不是完整的探索會話）
- **和** 過渡到變革創造

### 需求：引導工件建立

此技能應引導使用者完成每個工件，並透過旁白解釋其目的。

#### 場景：用旁白改變創作

- **何時** 建立變更目錄
- **然後** 代理程式執行 `openspec new change "<name>"` 帶有衍生的 kebab-case 名稱
- **AND** explains what a "change" is (container for thinking and planning)
- **AND** 顯示已建立的資料夾結構
- **並且** 在繼續之前暫停以等待用戶確認

#### 場景：帶旁白的提案建立

- **何時** 建立proposal.md
- **然後** 代理解釋了我們進行此更改的原因的提案
- **和** 根據選定的任務起草提案
- **並且** 在儲存之前向使用者顯示草稿以供批准
- **AND** 解釋了這些部分（原因、變化內容、功能、影響）

#### 場景：透過旁白創造規格

- **何時** 建立規格文件
- **THEN** agent explains specs define WHAT we're building in detail
- **AND** 解釋了需求/場景格式
- **並且** 根據提案功能建立規格文件
- **並且**指出規範成為保持同步的文檔

#### 場景：帶有旁白的設計創作

- **何時** 建立 design.md
- **然後** 代理解釋了設計捕獲我們將如何構建它
- **AND** 指出這是技術決策和權衡的地方
- **和** 對於小的變化，承認設計可能很簡短
- **並且** 根據提案和規格建立設計

#### 場景：建立帶有旁白的任務

- **何時** 建立tasks.md
- **然後** 代理解釋任務將工作分解為核取方塊
- **AND** 解釋了這些驅動應用階段
- **AND** 根據設計和規格產生任務列表
- **AND** 顯示任務並詢問是否準備好實施

### 要求：指導實施

該技能應透過連接回工件的敘述來執行任務。

#### 場景：帶敘述的實施

- **何時** 執行任務
- **然後** 代理在執行每項任務之前宣布該任務
- **AND** 實作程式碼庫中的更改
- **和**偶爾會參考規格/設計如何做出明智的決策
- **AND** 將每個任務標記為完成
- **並且**保持敘述輕鬆（不過度解釋）

#### 場景：實施完成

- **何時** 所有任務完成
- **然後** 代理宣布完成
- **和**總結了所做的事情
- **並且** 過渡到存檔階段

### 要求：存檔並附說明

此技能應存檔已完成的變更並解釋發生的情況。

#### 場景：帶有旁白的存檔

- **何時** 歸檔更改
- **然後** 代理說明存檔將變更移至指定日期的資料夾
- **AND** 執行存檔進程
- **AND** 顯示存檔變更的存放位置
- **AND** 解釋了長期價值（稍後找到決策）

### 要求：回顧和後續步驟

該技能應以回顧和命令參考結束。

#### 場景：最終回顧

- **何時** 入職完成
- **然後** 代理程式總結已完成的工作流程階段
- **AND** 強調這種節奏適用於任何尺寸變化
- **AND** 提供指令參考表（/opsx:explore、/opsx:new、/opsx:ff、/opsx:continue、/opsx:apply、/opsx:verify、/opsx:archive）
- **AND** 建議下一步操作（嘗試 /opsx:new 或 /opsx:ff 進行某些操作）

### 要求：優雅的退出處理

此技能將處理想要中途停止的用戶。

#### 場景：使用者想要停止

- **何時** 使用者表示他們想要在入職期間停止
- **然後** 代理人優雅地承認
- **AND** 指出正在進行的變更已儲存
- **AND** explains how to continue later with `/opsx:continue <name>`
- **AND** 無壓力退出

#### 場景：使用者只想快速參考

- **何時** 用戶說他們只想檢視命令
- **那麼** 代理提供命令備忘單
- **並且** 優雅地退出並鼓勵嘗試 `/opsx:new`
